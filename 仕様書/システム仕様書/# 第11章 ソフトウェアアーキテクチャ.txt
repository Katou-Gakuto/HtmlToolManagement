# 第11章 ソフトウェアアーキテクチャ

## 11.1 概要

本章では、本ソフトウェア全体の構成と各システムの責務を定義する。

実装方法ではなく、「どのシステムが何を担当するか」を明確にすることを目的とする。

本ソフトは高い拡張性・保守性・再利用性を重視した構造とする。

---

# 11.2 全体構成

ソフトウェアは以下の3つのプロジェクトで構成する。

```

設定プロジェクト
        │
        ▼
設定JSON
        │
        ▼
ゲームプロジェクト
        │
        ▼
Result
        │
        ▼
実行プロジェクト

```

それぞれは独立したプロジェクトとして実装する。

---

# 11.3 システム構成

ゲームプロジェクトは以下のシステムで構成する。

- Core
- Game
- UI
- FileSystem
- Action
- Pattern
- Skill
- Save
- Replay
- Statistics
- Plugin
- Audio
- Effect
- Input

各システムは独立して動作する。

---

# 11.4 Core

ゲーム全体を管理する。

役割

- 初期化
- 更新
- 終了処理
- システム管理

他システムの管理のみ行う。

---

# 11.5 Game

ゲームルールを管理する。

役割

- ゲーム状態
- スコア
- コンボ
- Focus
- Flow
- Rhythm

ゲームロジックのみ管理する。

---

# 11.6 FileSystem

ゲーム内ファイル情報を管理する。

役割

- File
- Folder
- OrganizeFolder
- DeleteFolder

実ファイル変更は行わない。

---

# 11.7 Action

ActionQueueを管理する。

役割

- Action追加
- Action更新
- Action削除
- Undo
- Redo

---

# 11.8 Pattern

Patternを管理する。

役割

- 読込
- 保存
- 検索
- 提案

---

# 11.9 Skill

Skillを管理する。

役割

- 検索
- Skill Chain
- 条件管理
- プリセット

---

# 11.10 Save

保存システム。

役割

- JSON保存
- JSON読込
- Version管理
- オートセーブ

---

# 11.11 Replay

Replayを管理する。

役割

- 記録
- 保存
- 再生

---

# 11.12 Statistics

統計を管理する。

役割

- プレイ時間
- 使用回数
- ベスト記録

---

# 11.13 Plugin

Pluginを管理する。

役割

- 読込
- Version確認
- 有効・無効

---

# 11.14 UI

ゲームUIを管理する。

役割

- HUD
- メニュー
- ログ
- 整理View
- ミニマップ

UIのみ管理する。

---

# 11.15 Audio

音を管理する。

役割

- BGM
- 効果音
- 音量

---

# 11.16 Effect

演出を管理する。

役割

- パーティクル
- Flow演出
- コンボ演出
- Pattern Burst

---

# 11.17 Input

入力を管理する。

対応デバイス

- キーボード
- マウス
- タッチ

キー設定はJSONから取得する。

---

# 11.18 システム依存

システム間の依存を最小限にする。

例

UI

↓

Action

↓

FileSystem

のような直接依存を避け、

必要に応じてManagerやイベントを介して連携する。

---

# 11.19 イベント駆動

各システムはイベント通知によって連携する。

例

Action追加

↓

UI更新

↓

ログ追加

↓

スコア更新

直接呼び出しを極力減らす。

---

# 11.20 データと表示の分離

ゲームデータとUIを分離する。

データ変更時、

UIは通知を受けて更新する。

UIからゲームデータを直接変更しない。

---

# 11.21 将来拡張

本アーキテクチャは以下の拡張を考慮する。

- マルチプラットフォーム
- オンライン機能
- クラウド同期
- Marketplace
- AI支援
- 新しいPlugin
- 新しいPack

システム追加時も既存構造を大きく変更しないことを前提とする。
